# Repository Guidelines

## Project Structure & Module Organization

This is a Rust binary project for nannou creative-coding sketches.

- `Cargo.toml` defines the workspace members and shared profiles.
- `Cargo.lock` is committed for reproducible local runs.
- `rust/crates/main/main.rs` contains the thin nannou app runner, window setup, fullscreen shortcut, and persistence.
- `rust/crates/hot-reload/lib.rs` contains the hot-reloadable sketch library: model, update, and drawing code.
- `rust/crates/mcp-server/main.rs` contains a read-only MCP server built with the official Rust SDK.
- `scripts/Run.ps1` runs the app from the repository root.
- `target/` is generated by Cargo and must remain untracked.

Place runner code under `rust/crates/main/` and hot-reloadable art code under `rust/crates/hot-reload/`. Put images, fonts, shaders, or other runtime files under `assets/`.

## Build, Test, and Development Commands

- `cargo check` verifies compilation without producing a runnable binary.
- `cargo build` compiles the app.
- `cargo run -p nannou-creative-coding` launches the nannou window.
- `.\scripts\Install.ps1` installs or updates Rust through rustup.
- `.\scripts\Run.ps1` launches the app through the repository script.
- `.\scripts\RunWithHotReload.ps1` keeps the app running and hot reloads `rust/crates/hot-reload/lib.rs`.
- `.\scripts\RunMcpServer.ps1` runs the read-only MCP stdio server.
- `cargo fmt --check` verifies Rust formatting.
- `cargo fmt` applies standard Rust formatting.

Run commands from the repository root.

## Coding Style & Naming Conventions

Use standard Rust formatting via `rustfmt`. Prefer descriptive snake_case names for functions and variables, PascalCase for types, and SCREAMING_SNAKE_CASE for constants such as `WINDOW_WIDTH`.

Keep sketch constants near the top. Use conventional nannou lifecycle functions: `main` and window setup in `rust/crates/main/main.rs`, with hot-reloadable `update` and `view` functions in `rust/crates/hot-reload/lib.rs`.

## Testing Guidelines

There are no automated tests yet. For compile-level validation, run:

```powershell
cargo fmt --check
cargo check
```

If logic grows beyond simple drawing setup, add unit tests in the relevant Rust module or integration tests under `tests/`. Name tests after behavior, for example `centers_window_from_monitor_bounds`.

## Commit & Pull Request Guidelines

Use concise imperative commit messages, for example `Add centered nannou starter sketch`.

Pull requests should include a short change summary, verification run, and screenshots or visual notes for drawing/window changes.

## Agent-Specific Instructions

Keep changes scoped to this repository. Do not commit build output or local machine configuration. Prefer updating `scripts/Run.ps1` when the run workflow changes.

Users may describe drawings and art in natural language. Translate requests into nannou sketches using the official guide at `https://guide.nannou.cc/`, especially drawing, color, geometry, animation, and app-structure patterns. Put sketch edits in `rust/crates/hot-reload/lib.rs` so `RunWithHotReload.ps1` can rebuild the hot library.

Window behavior belongs in `rust/crates/main/main.rs`. `F` toggles fullscreen and the runner persists fullscreen state, monitor, position, and size to `target/window-state.json`.

MCP helper behavior belongs in `rust/crates/mcp-server/main.rs`. Keep it read-only unless the user explicitly asks for write-capable MCP tools.

## Git Safety

Non-destructive Git operations are allowed for normal repository work. This includes `git init`, creating a new remote repository when explicitly requested, adding remotes, creating branches, staging, committing, fetching, and normal non-force pushes.

Destructive Git remains prohibited. Do not rewrite history, discard local changes, delete branches or tags, force-push, run `git reset --hard`, run `git clean`, rebase, amend, squash, or delete repositories.
